# `before/` — naive or problematic code

Put the **baseline** here: the smallest example that shows why the pattern matters.

- Prefer **one focused story** (one package or a few files), not a full application.
- Name types clearly so `after/` can show the same scenario refactored.
- A short note at the top of the main file (`// Smelly: …`) helps reviewers and learners.

If this folder is empty, explain in the pattern’s main `README.md` why (rare).
